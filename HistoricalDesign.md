## 1. 第一阶段：原始设计理念

### 1.1 设计动机

在早期 RK 裸机框架中，`rk_task` 提供了最基础的协作式多任务能力——任务函数在主循环中被轮询调用。其本质是一个函数指针链表，每次轮到某任务时完全执行一遍然后返回。这种方式存在以下根本限制：

- **无抢占能力**：长任务阻塞整个系统，无法保证实时性
- **无延时/阻塞语义**：任务只能通过 `return` 让出 CPU，无法"挂起自己 N 个 tick 后恢复"
- **无事件同步机制**：任务间通信依赖全局变量加轮询，无标准 wait/post 模型
- **无独立栈**：所有任务共享调用栈，局部变量相互污染，任务函数无法深度嵌套

### 1.2 技术来源

**T10OS 事件驱动模型**：借鉴了事件驱动理念——每个任务本质上等待一组事件标志，事件满足时被唤醒执行，执行完毕或主动挂起时回到等待状态。这一模型天然适合嵌入式系统中"事件到来→处理→继续等待"的典型工作负载。但 T10OS 将所有任务统一为事件驱动模型，对纯轮询型的后台任务（如 LED 闪烁、看门狗喂狗）显得过重。

**FreeRTOS PendSV 机制**：FreeRTOS 利用 ARM Cortex-M 的 PendSV 异常实现了上下文切换的"延迟执行"：在 SysTick 或某任务主动请求调度时，仅触发 PendSV（设置 PendSV 挂起位），**不立即切换**。PendSV 被编程为最低优先级异常，在所有其他异常处理完毕后才会执行，从而避免在中断处理中途进行上下文切换导致优先级反转。rktask 继承了这一核心设计，并精简到极致。

### 1.3 设计目标

| 准则 | 目标 | 验证方式 |
|:----|:----|:---------|
| **最高性能** | PendSV 切换 ≤ 15 条指令 | 反汇编计数 |
| **最小开销** | 内核 Flash ≤ 1.5 KB, RAM ≤ 200 B（不含任务栈） | 编译后 map 文件 |
| **最简单** | 仅两种任务类型，无优先级数字 | API 头文件中的类型定义 |

为什么这样定：
- PendSV 15 条指令限制意味着一次上下文切换大约 30-50 ns（以 72 MHz STM32 计），接近理论极限
- 1.5 KB Flash 的约束逼迫每行汇编/每字节 RAM 都必须有明确用途——绝不包含 FreeRTOS 中"可能有用"的可选特性
- "两种类型无优先级"消除了 RTOS 学习中最困惑的优先级翻转、优先级继承等概念

### 1.4 两类型任务架构

```
COOP    协作式 — 不可被抢占，执行完整原子操作
        适合：协议解析、状态机、Flash 编程、命令处理

PREEMPT 抢占式 — 时间片轮转，可被自动剥夺
        适合：周期性采集、通信轮询、软件定时器
```

**COOP 优先策略**：只要 COOP 就绪链表非空，PREEMPT 任务完全得不到 CPU。COOP 任务通过 `rk_yield()` 主动让出，让出频率由开发者精确控制。PREEMPT 任务在 COOP 全部阻塞或 yield 后才能运行。这一策略避免了传统 RTOS 中"低优先级任务饿死"的问题——系统中没有优先级数字，只有类型归属。

### 1.5 原始数据结构设计

#### TCB（设计规格：32 字节，双向链表）

```c
typedef struct rk_tcb {
    uint32_t    *sp;            // 栈指针 (PSP)
    struct rk_tcb *next;        // 链表后继指针
    struct rk_tcb *prev;        // 链表前驱指针 (双向)
    uint32_t    delay_remain;   // 剩余延时 tick 数
    uint32_t    evt_wait;       // 正在等待的事件标志位掩码
    uint32_t    evt_rcv;        // 已收到的事件标志位掩码
    uint8_t     evt_mode;       // 等待模式
    uint8_t     type;           // 任务类型
    uint8_t     state;          // 状态
    uint8_t     reserved;       // 对齐填充
} rk_tcb_t;
```

#### 就绪链表：两条独立的循环双向链表

```
ready_coop:     COOP 类型就绪任务的循环双向链表
ready_preempt:  PREEMPT 类型就绪任务的循环双向链表
```
- 双向特性允许 O(1) 删除——当任务从 BLOCKED 唤醒时，不需要遍历链表
- 通过链表旋转算法实现公平轮转

#### 延时链表：Delta 编码排序链表

按剩余延时值升序排列，每个节点存储相对前驱的差值。SysTick 中只需对链表头节点递减，实现 O(1)/tick。

#### 事件组：

```c
typedef struct rk_evt {
    uint32_t    flags;          // 事件标志位
    rk_tcb_t    *list;          // 等待此事件组的任务链表 (单向)
} rk_evt_t;
```

#### 临界区：BASEPRI 机制

选择 BASEPRI 而非 PRIMASK，理由：
- SysTick 不丢失：临界区内 SysTick 仍可触发
- PendSV 延迟执行：自动实现"延迟切换"
- 天然嵌套：保存/恢复 BASEPRI 支持嵌套临界区
- 实时性更优：高优先级中断不受临界区影响

### 1.6 上下文切换设计

PendSV_Handler 设计为 13 条指令（M3+），流程：

```
1. mrs r0, psp            读取当前 PSP
2. stmdb r0!, {r4-r11}    保存 R4-R11
3. ldr/ldr/str            更新 TCB->sp
4. bl rk_sched            选择下一个任务
5. ldr/ldr                加载新 TCB->sp
6. ldmia r0!, {r4-r11}    恢复 R4-R11
7. msr psp, r0 / bx lr    设 PSP 并返回
```

## 2. 第二阶段：初始实现与设计差异

### 2.1 实际实现与设计规格的差异

在代码审查（P1-13）中发现：设计文档描述的是一个**与代码有重大差异**的架构。以下列出所有已知差异：

| 差异项 | 设计文档描述 | 初始代码实际情况 |
|:------|:-----------|:---------------|
| **链表类型** | 循环双向链表 | 单向链表（无 `prev` 指针） |
| **TCB 大小** | 32 字节（含 `prev`、`evt_rcv`） | 40 字节（无 `prev`、`evt_rcv`，但有 `stack_base`、`stack_size`、`name`） |
| **延时队列** | Delta 编码，O(1)/tick | 无序链表，O(N)/tick 遍历全部 |
| **事件 API** | 指针式 `rk_evt_t *evt` 参数 | ID 式 `rk_id_t id` 参数 |
| **事件字段** | `evt_wait` + `evt_rcv` 分离 | `evt_mask` 单一字段 |
| **就绪链表** | 链表旋转算法 | 头移除/尾插入（pop/add） |
| **API 名** | `RK_TASK_COOP` / `RK_TASK_PREEMPT` | `RK_TYPE_COOP` / `RK_TYPE_PREEMPT` |
| **API 前缀** | `rk_` 短前缀 | 初始为 `rk_`，审查后改为 `rk_task_` |
| **任务参数** | `void *param` 参数传入 R0 | 无参数 |
| **栈保护** | 未提及 | `rk_stk_init` 在栈底写 `0xDEADBEEF` (审查后添加) |

### 2.2 差异产生原因

1. **双向链表 → 单向链表**：在任务数 ≤ 16 的场景下，单向链表 O(N) 移除与双向链表 O(1) 移除的实际性能差异可忽略，但代码复杂度和 TCB 尺寸（+4 字节）下降显著
2. **Delta 编码延迟队列 → 无序列表**：初始实现时更注重 API 接口的快速成型，性能优化（P1-1）被有意识地延期到 v2.0
3. **指针事件 API → ID 事件 API**：ID 式 API 更安全（可在调用点做边界检查），且无需处理指针悬空问题

### 2.3 初始代码已知问题（审查前）

完成初始实现后，代码存在以下已知或未知问题：

| 问题 | 严重性 | 后续修复 |
|:----|:------|:---------|
| `rk_lock()` 注释中 BASEPRI 屏蔽语义描述错误 | 误导 → 可导致误用 | P0-1 |
| 任务自我删除后 TCB 槽位泄漏 | 重复自删 16 次后系统崩溃 | P0-2 |
| TCB 布局缺少编译期断言 | 后继修改可能导致 PendSV 静默崩溃 | P0-3 |
| `rk_evt_wait` 超时路径丢失迟到事件 | 极端时序下事件信号丢失 | P1-9 |
| `rk_self()` O(N) 线性扫描 | 高频调用时性能浪费 | P1-3 |
| 线程 API 缺少 ISR 调用保护 | ISR 中误调用可导致未定义行为 | P1-4 |
| 无栈溢出检测 | 栈溢出静默破坏相邻内存 | P1-2 |
| 无栈重叠检测 | 两个任务使用重叠栈区 | P1-11 |
| 魔法数字约 15 处 | 难以维护 | P1-7 |
| Allman 风格违规约 70 处 | 风格不一致 | P1-6 |

### 3.3 P0 发现：致命缺陷

#### P0-1: SysTick 与高优先级 ISR 的链表竞态

**原始分析错误**：最初架构审查误以为 BASEPRI=0xF0 不屏蔽 SysTick（优先级 0xF0）。但 ARMv7-M 规范证明：BASEPRI 屏蔽**优先级值 ≥ 自身的所有异常**，因此优先级 0xF0 的 SysTick **确实被**每个 `rk_lock()` 临界区屏蔽。

**真正的竞态**：

```
1. SysTick_Handler 正在遍历延时链表
2. 优先级 0x40 的外部中断触发，抢占 SysTick
3. ISR 调用 rk_evt_post()，修改就绪/延时链表
4. ISR 返回后 SysTick 继续迭代，使用已损坏的 t->next 指针
→ 损坏的链表 → 孤立/双重调度任务 → HardFault
```

**修复**：(1) 修正注释，正确描述 BASEPRI 屏蔽语义 (2) `rk_evt_post()` 中添加 ISR 优先级守卫，拒绝优先级高于 SysTick 的 ISR 调用 (3) 更新头文件注解，明确 ISR 使用约束

#### P0-2: 自我删除导致 TCB 槽位永久泄漏

**原始分析错误**：架构审查最初诊断"PendSV 读取 TCB->sp=NULL 后 `stmdb r0!, {r4-r11}` 会导致 HardFault"。事实上 PendSV 保存阶段先用 `mrs r0, psp` 从硬件 PSP 读取，然后 `str r0, [r2]` **覆盖**了 NULL。真正的 bug 是槽位泄漏——自我删除后 `sp` 变为非 NULL（被 PendSV 写入有效 PSP 值），`rk_create` 的 `sp == NULL` 扫描无法重用该槽位。

**修复**：引入 zombie 状态机——自我删除时标记 `state=RK_STATE_DELETED`，`rk_create()` 扫描时回收此类槽位。

#### P0-3: TCB 布局缺少编译期断言

PendSV 汇编通过硬编码偏移 0 访问 `sp`。如有人前插字段，上下文切换将静默损坏栈指针，导致随机 HardFault。

**修复**：在结构体定义后添加 `_Static_assert(offsetof(rk_task_t, sp) == 0, ...)`。

#### P0-4: sp 字段 volatile 修饰错位

`uint32_t volatile *sp` 中 `volatile` 限定的是**指向的数据**，而非**指针值本身**。上下文切换需要的是指针值不被编译器缓存（跨 asm 边界）。

**修复**：改为 `uint32_t *volatile sp`。

### 4.2 性能优化 (OPT-1 ~ OPT-3)

在正确性修复完成后，实施了三项性能优化，优先级策略为**性能 > RAM > Flash**：

#### OPT-1: Delta 编码延时队列 (SysTick O(N)/tick → O(1)/tick)

**延时前**：每毫秒遍历全部延时任务逐个递减
**延时后**：每毫秒只递减头节点一个，到期批量弹出

新增函数 `rk_task_delay_insert()` 和 `rk_task_delay_remove()`，改动点涉及 `rk_delay`、`rk_evt_wait`、`rk_evt_post`、`SysTick_Handler`、`rk_evt_del`。

```
16 任务全延时场景：
  修改前: 16 次递减 + 16 次比较 = ~50 指令/ms
  修改后: 1 次递减 + 1 次比较 = ~5 指令/ms
```

**RAM**: 0 字节 | **Flash**: +60 字节

#### OPT-2: 数组化就绪链表 (消除分支，简化代码)

```
修改前: coop_head/tail + preempt_head/tail (6变量, 每调用点 if/else 分支)
修改后: ready_head[2] + ready_tail[2] (4变量, 按 t->type 索引)
```

新增辅助函数 `ready_add()` 和 `ready_rem()`，约 8 处调用点从分支变为数组索引。

**RAM**: 0 字节 | **Flash**: -80 字节

#### OPT-3: PendSV 返回新任务 sp 在 r0 (13 指令 → 9 指令)

`rk_sched()` 签名从 `void rk_sched(void)` 改为 `uint32_t *rk_sched(void)`，直接返回新任务 `sp` 值在 r0 中，PendSV 无需再重新加载 `rk_cur` 并解引用。

```
修改前 (13 指令):  stmdb → 更新 TCB → push/pop → bl → ldr×2 → ldmia
修改后 (9 指令):   stmdb → 更新 TCB → bl → ldmia
```

**RAM**: 0 字节 | **Flash**: -8 字节 | **每次切换省 ~4 周期**

#### 三次优化合计性能影响

| 指标 | 优化前 | 优化后 |
|:----|:------|:------|
| SysTick 延时处理 | O(N)/tick (N=任务数) | **O(1)/tick** |
| PendSV 指令 (M3+) | 13 条 | **9 条** |
| 就绪链表选择 | if/else 分支 (每调用点) | 数组索引 (零分支) |
| `rk_self()` | O(N) 线性扫描 | **O(1)** |
| PendSV Flash 占用 | 基准 | **-8 字节** |
| 总代码 Flash | 基准 | **-28 字节** (净减) |
